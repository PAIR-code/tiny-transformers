Relational Linear Logic Engine

Welcome to the semantics and documentation guide. This page explains the core concepts, syntax rules, and evaluation model of the linear logic engine, highlighting how term matching and function evaluation work together.

help_outline

1. What is Linear Logic?

In classical logic, assertions represent truth values that are static and can be duplicated or discarded at will. In contrast, Linear Logic is a substructural logic where hypotheses represent physical resources.

info
The Resource Rule: Every resource must be consumed exactly once in a state transition. You cannot duplicate a resource unless explicitly allowed by a rule, and you cannot let a resource vanish unless it is explicitly consumed.

This makes linear logic uniquely suited for state tracking, planning puzzles (e.g. the River Crossing puzzle), and protocol channel simulations, where actions consume preconditions and produce postconditions.

3. Pattern-Matching Functions

        

        

          

            Functions are declared using the fun keyword and support pattern-matching clauses separated by 
            the | pipe character. Logic variables in clause patterns are prefixed with a question mark (?).
          

          
          

            
Pattern-Matched Opposite Bank

            

          


          

            When a function is called, the evaluator attempts to match the arguments structurally against the 
            patterns of each clause from left to right. The first clause that matches binds its variables and evaluates 
            its body.

4. State-Transition Actions

        

        

          

            Actions represent the transitions in our linear resource multiset. They are defined using the 
            action keyword. An action consumes resources on the Left-Hand Side (LHS) of the 
            -o (Lolli) operator and produces resources on the Right-Hand Side (RHS).
          


          

            
Lolli Action Syntax

            

          


          

            LHS resources act as preconditions. During matching, the engine finds a subset of active linear 
            resources in the story that matches the structural patterns on the LHS. Once applied, those resources 
            are removed from the multiset, and the RHS resources are introduced.

5. How Matching & Evaluation Interact

        

        

          

            A core feature of this engine is the interaction between pattern-matching unification (on the LHS) 
            and recursive evaluation (on the RHS).
          


          
The Evaluation Model: Call-by-Value (CBV)

          

            Terms in the engine are evaluated using a strict Call-By-Value reduction scheme. This means before any 
            constructor or function body is processed, all of its arguments are recursively evaluated to normal form.
          


          
LHS Matching operates on Normal Forms

          

            Crucially, all active resources inside the multiset are always kept in normal form (fully evaluated and reduced terms).
            Unification and pattern matching on the LHS of a rule do not evaluate function calls during matching; instead, they structurally compare the resource against the pattern.
            Because both initial resources (loaded from context) and newly produced resources (added after RHS transition) are recursively reduced to normal form first, structural patterns (like suc(?x)) match successfully.
          


          

            info
            

              Example of Normal Form Matching: 
              If the initial context defines _r1: add(suc(0), suc(0));, it is immediately evaluated to suc(suc(0)).
              When matching rule decrement: {{ '{' }} ?x: suc(?v) {{ '}' }} -o {{ '{' }} ?y: ?v {{ '}' }}, structural matching succeeds because it unifies pattern suc(?v) against evaluated term suc(suc(0)) (binding ?v to suc(0)).
              If the resource remained as the unevaluated term add(suc(0), suc(0)), structural unification would fail.
            

          


          
Step-by-Step Transition Lifecycle

          

            

              Step A: LHS Resource Unification
              
                The engine matches LHS patterns against active resources. Active resources are *always* in normal form. 
                Unification binds logic variables (e.g. ?from) to their matched values.
              
            

            

              Step B: RHS Substitution
              
                The variable bindings synthesized from LHS matching are substituted into the RHS patterns.
              
            

            

              Step C: CBV Reduction & Function Application
              
                Before the new resources are added to the multiset, the substituted RHS terms are fully evaluated 
                and reduced to normal form. Any function applications are executed.
              
            

            

              Step D: Multiset Transition
              
                The matched LHS resources are deleted, and the freshly evaluated RHS resources are added to the state.
              
            

          


          
A Concrete Trace: Peano Addition

          

            Suppose the current context has resource _r1: counter(suc(0)) (value 1), and we trigger the 
            following action:
          

          

            
LHS & RHS Flow

            

          


          

            

              1. Match
              
                LHS pattern counter(?c) matches resource counter(suc(0)). 
                Variable ?c binds to suc(0).
              
            

            

              2. Substitute
              
                Substitution into RHS yields: counter(add(suc(0), suc(0))).
              
            

            

              3. Evaluate Arguments (CBV)
              
                Evaluate argument add(suc(0), suc(0)). Since add is a function, we match 
                against its clauses:
              
              add(suc(?x), ?y) = suc(add(?x, ?y))
              
                Here, ?x binds to 0 and ?y binds to suc(0). 
                This reduces to: suc(add(0, suc(0))).
              
            

            

              4. Evaluate Nested Clause (CBV)
              
                Evaluate argument of suc, which is add(0, suc(0)). Matches clause:
              
              add(0, ?y) = ?y
              
                This yields: suc(0).
              
            

            

              5. Final normal form
              
                Outer constructor evaluates to suc(suc(0)). The RHS resource becomes counter(suc(suc(0))).
